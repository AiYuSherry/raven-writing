# 女娲 · Skill 造人术与项目蒸馏

女娲不是复制人，也不是简单摘要。它把可复用的能力提炼成一套能被 Agent 调用的操作系统：

- 人物/主题：提炼心智模型、决策启发式、表达 DNA、反模式和诚实边界。
- 学术作者：提炼问题意识、思想谱系、概念词库、论文结构、论证框架、注释习惯和写作质检规则。
- 项目/仓库：提炼项目类型、核心命令、使用流程、故障处理和验证方法。
- 模糊需求：先诊断用户真正需要哪类思维框架或项目能力，再决定蒸馏对象。

## 入口分流

收到任务后先判断路径。

| 用户输入 | 路径 | 默认动作 |
| --- | --- | --- |
| 明确人名、组织、主题 | 人物/主题路径 | 确认对象和用途，创建 perspective skill |
| 学术作者、论文作者、法学/社科写作者 | 学术作者路径 | 基于论文/著作语料创建 academic author skill |
| 模糊困惑或能力诉求 | 诊断路径 | 最多追问 2 轮，推荐 2-3 个候选 |
| GitHub URL | 项目路径 | 运行项目分析，生成 project skill |
| 本地项目目录 | 项目路径 | 读取本地 README、配置和源码结构，生成 project skill |
| 更新已有 skill | 更新路径 | 先读取旧 SKILL.md，再补调研和修订 |

用户说「就做 XX」且对象明确时，不要过度追问。默认：全面画像、作为思维顾问、无本地语料；如果是 GitHub URL，默认生成项目使用型 skill。

如果用户明确说「学术作者蒸馏」「论文作者蒸馏」「实现文风、思想、论文撰写的蒸馏化」「用他的论文能力写新文章」，优先走学术作者路径。该路径不默认角色扮演本人，而是生成可用于新论文构思、论证展开和风格化改写的写作模型。

## 人物/主题路径

### Phase 0: 澄清

只确认真正会影响产物的点：

1. 对象是谁，是否存在同名歧义。
2. 用途是思维顾问、决策参考、内容创作视角，还是角色扮演。
3. 是否有本地一手素材，例如 PDF、访谈 transcript、视频字幕、博客导出、内部文档。
4. 是新建还是更新已有 skill。

如果用户没有本地素材，走网络搜索；如果用户说「只用我给的素材」，不要联网补充。

### Phase 1: 调研

在目标 skill 内创建自包含目录：

```text
[target-skill]/
├── SKILL.md
├── scripts/
└── references/
    ├── research/
    │   ├── 01-writings.md
    │   ├── 02-conversations.md
    │   ├── 03-expression-dna.md
    │   ├── 04-external-views.md
    │   ├── 05-decisions.md
    │   └── 06-timeline.md
    └── sources/
```

调研维度：

- `01-writings.md`：著作、长文、论文、系统表达。
- `02-conversations.md`：访谈、播客、AMA、追问下的即时反应。
- `03-expression-dna.md`：高频词、句式、幽默方式、禁忌词、语气。
- `04-external-views.md`：他人评价、批评、争议和同行对比。
- `05-decisions.md`：关键决策、转折点、事后复盘。
- `06-timeline.md`：完整时间线和最近 12 个月动态。

硬规则：

- 调研必须写入目标 skill 的 `references/research/`，不存文件等于没调研。
- 明确区分「他说过的」「别人说他的」「我推断的」。
- 保留矛盾，不要为了顺滑而消除矛盾。
- 中国人物优先用原始视频、播客、公开采访、权威中文媒体；排除知乎、百度百科和不可验证的微信公众号转述。

需要更细的抽取框架时读取 `references/extraction-framework.md`。生成人物 skill 时读取 `references/skill-template.md`。

## 学术作者路径

当用户要蒸馏论文作者、法学家、社科学者或其他严肃学术写作者时，走本路径。不要把学术作者 skill 写成普通人物扮演，也不要把论文文风改造成公众号、博客或营销文。

### Phase A0: 语料分层

先区分语料身份：

- 一手写作样本：作者本人论文、著作章节、访谈自述、演讲稿。
- 语境材料：同场对话、他人论文、评论、书评、编者按、访谈中的提问。
- 身份补证：学校官网、期刊页面、出版社页面、权威机构简介。

硬规则：

- 只有一手写作样本可以进入文风、句式、注释习惯和论文结构统计。
- 语境材料只能用于理解问题域、学术对话和外部评价，不得混入作者表达 DNA。
- 无法确认作者身份的文件必须标注为 `context_only` 或 `uncertain`。

### Phase A0.5: 写作对象研究

如果生成的学术作者 skill 将用于写新文章，必须为它设计“先研究对象，再调用作者模型”的工作流。作者模型不能替代对象事实。遇到 OpenClaw、某个技术、制度、公司、案例、政策、新近事件或法律风险议题时，默认需要实时检索或本地核验；只有用户明确说“只用本地材料”时，才跳过联网并写明资料边界。

法学、社科和政策类写作还必须加入“专业知识库/私有知识库层”。在完成网络/官方事实检索后，默认进入用户绑定的 ima 知识库检索相关论文和学术观点；若用户有指定知识库，优先使用指定知识库，若未指定，法学数字治理主题优先搜索「数字法学研究」。知识库结果只作为学术文献线索和观点来源，不能替代规范原文、官方事实或本地核验。

对象研究必须输出到目标 skill 的 `references/research/object-research-dossier.md` 或在 `SKILL.md` 中要求每次写作前临时生成同名研究档案。档案至少包括：

- 对象类型：技术/项目、制度/政策、人物/组织、案例/事件、法律风险。
- 来源优先级：官方文档、法律法规、监管文件、司法案例、GitHub/README、CLI 帮助、本机状态、专业知识库/私有知识库论文、权威媒体或学术材料。
- 事实层：技术原理、制度结构、关键行为链、版本/时间、利益相关方。
- 文献层：ima 等知识库中的相关论文、摘要/高亮/片段、既有观点、理论争点和研究缺口。
- 风险层：数据、权限、身份、网络安全、知识产权、证据、专业责任、组织授权等。
- 不确定点：无法核验、时效性强、仅有二手信息或可能变化的事实。

技术/项目类对象优先查官方文档、GitHub/README、CLI 帮助和本地运行状态；法律/政策类对象优先查法律法规、监管文件、司法案例和权威机构页面；公司/组织类对象优先查官网、公告、监管披露和可信新闻；案例/事件类对象必须标注时间线和来源差异。

需要细则时读取 `references/object-research-protocol.md`。生成学术作者 skill 时，模板中必须包含“事实来自对象研究，文献观点来自专业知识库/学术材料，论证方式来自作者蒸馏”的三层分离规则，并明确没有完成对象研究和学术文献检索，不得直接套作者文风写作。

### Phase A0.75: 样本文笔对照层

学术作者 skill 不能只迁移问题意识、思想谱系或概念密度。凡目标 skill 将被用于“用某作者写”“某作者风格写论文”“按某作者论文方式改写”，必须内置强制的样本文笔对照层。

生成 academic author skill 时，必须写入以下规则：

- 调用者点名作者风格时，先读取或抽样一手写作样本，再生成正文；只读 `SKILL.md` 概述不合格。
- 至少抽样 2 篇 `primary_writing`，优先同题域样本，再补一篇方法论/理论支架样本；不可用时读取已抽取的一手样本研究文件。
- “样本文笔对照笔记”必须作为生成前置产物形成，不能只是可选建议，也不能在正文完成后倒填；没有该笔记，不得进入正文、改写或续写。
- 如果找不到可读取的一手样本文本，也找不到已抽取的一手样本研究文件，目标 skill 必须要求暂停完整成文；只能输出资料边界、待补样本清单、提纲或研究计划。用户要求继续时，也必须显著标注“未完成样本文笔对照”的边界。
- 样本文笔对照笔记必须记录句法观察、概念链、段落节奏、常用论证动作、禁止迁移和本次改写指令，并说明这些观察如何约束正文段落。
- 明确浅层合规失败模式：只做 IMA/知识库检索、只做对象研究、只套问题意识、只提高概念密度、只写“问题像”但句法和段落不像，均不算严格执行。
- 写入“失败产出反例处理”：类似《我们真的有个人信息同意撤回权么？》这类概念密度高、问题意识方向近似，但句法、段落节奏、概念链和作者辨识度不像的产出，quality gate 必须判失败。
- 写入“低相似度警报”：草稿若像普通 AI 学术评论、政策解读、课堂作业、公众号文章或“法条+建议”模板，必须重新抽样并重写。
- 写入生成后退回规则：检查样本文笔对照是否为前置产物、句法重写、段落节奏校准、概念链连续性、事实/文献/作者模型三层分离；不通过则不得交付。

### Phase A1: 学术抽取

在目标 skill 内创建：

```text
[target-skill]/
├── SKILL.md
└── references/
    ├── research/
    │   ├── corpus-manifest.md
    │   ├── problem-consciousness.md
    │   ├── concept-lexicon.md
    │   ├── argument-patterns.md
    │   ├── paper-structure-patterns.md
    │   ├── expression-dna.md
    │   ├── style-comparison-notes.md
    │   ├── object-research-dossier.md
    │   └── source-boundaries.md
    └── examples/
        ├── topic-outline-sample.md
        └── rewrite-sample.md
```

抽取维度：

- 问题意识：作者反复进入问题的入口、批判对象、被遮蔽的真实问题和默认关怀。
- 思想谱系：作者常调用的理论传统、关键学者、比较法对象、社会理论和政治哲学资源。
- 概念词库：高频概念、概念组合、价值判断词、制度分析词、反复出现的分类轴。
- 论证框架：如何破除流行分类，如何建立新类型学，如何连接规范论证与制度事实。
- 论文结构：题名、摘要、引言、章节推进、结论、脚注和材料组织习惯。
- 表达 DNA：句长、限定语、转折、设问、判断强度、抽象与案例的切换节奏。
- 样本文笔对照：从一手样本提取句法、概念链、段落节奏和论证动作，形成调用时可复用的对照笔记模板。
- 反模式：这种作者不会如何开头、如何下判断、如何引用、如何写结论；特别标出“只迁移思想/问题意识但文笔不像”的失败模式。

需要细则时读取 `references/academic-author-distillation.md`。生成学术作者 skill 时读取 `references/academic-author-template.md`。

### Phase A2: 提炼为写作系统

学术作者 skill 必须能执行下列任务：

- 诊断一个新题目是否适合用该作者模型处理。
- 生成论文题目、摘要、关键词和三级提纲。
- 写引言：从既有范式的问题开始，进入作者式问题意识。
- 展开章节：用作者常用的分类、比较、反驳和重构方式组织论证。
- 改写草稿：让普通学术段落更接近该作者的论证节奏和表达密度。
- 自检：指出哪些句子像泛 AI 法学写作，哪些地方偏离该作者模型。

### Phase A3: 生成与验证

学术作者产物必须：

- 明确声明「基于公开/本地材料提炼，不代表本人观点」。
- 禁止冒充作者本人、禁止署作者名、禁止声称这是作者真实新作。
- 保留语料边界、调研日期和不能确定之处。
- 借鉴写作方法而不照搬长段原文。
- 内置样本文笔对照层、文笔相似度硬约束、低相似度警报、样本缺失暂停规则和退回重写规则。
- 用分层质检检查硬规则、样本文笔对照前置产物、样本缺失边界、失败产出反例、风格一致性、论证密度和作者辨识度。

### Phase 2: 提炼

把调研压缩成可运行的认知操作系统：

- 3-7 个核心心智模型：名称、证据、适用场景、失效条件。
- 5-10 条决策启发式：具体到可执行判断，而不是抽象品质。
- 表达 DNA：句式、节奏、词汇、引用习惯、幽默和确定性。
- 反模式：这个视角绝不会怎么想、怎么说、怎么做。
- 诚实边界：公开信息局限、过时风险、争议点和不可替代本人之处。

### Phase 3: 生成与验证

生成的 skill 必须：

- `SKILL.md` 开头有 `name` 和 `description` frontmatter。
- 触发描述清楚，用户用自然语言也能命中。
- 文件自包含，复制整个目录即可独立使用。
- 不把长篇调研塞进主 `SKILL.md`，长材料放入 `references/`。
- 不留下 `TODO`、`TBD`、`xxx`、`待补充` 等占位符。

可运行：

```bash
python3 scripts/quality_check.py /path/to/target-skill
```

如果脚本不适配当前产物，至少手动检查 frontmatter、引用路径、占位符和调研文件完整性。

## 项目/GitHub 路径

当用户给出 GitHub URL、本地项目目录，或说「蒸馏这个项目/仓库为 skill」时，走项目路径。

### Phase P1: 项目分析

GitHub URL 默认运行：

```bash
bash scripts/analyze-repo.sh <github-url> /tmp/skill-gen-<repo-name>
```

然后读取：

- `/tmp/skill-gen-<repo-name>/project-profile.json`
- 克隆目录里的 README
- `docs/`、`examples/`、核心配置文件和 CLI/API 入口

本地项目则直接从用户给的目录分析，不要重复 clone。

### Phase P2: 类型适配

根据项目类型决定 skill 重点：

- CLI：安装、命令速查、参数组合、输出格式、批处理场景。
- Library：API 用法、集成模式、返回值、版本/类型兼容。
- Framework：核心概念、架构、扩展点、项目脚手架。
- Web app：本地开发、部署、环境变量、端口和数据依赖。
- DevOps：平台支持、前置条件、配置定制、回滚和排错。
- Unknown：先提炼最小可验证工作流，再补充项目类型判断。

更完整的项目蒸馏规则见 `references/project/project-distillation.md`。

### Phase P3: 生成项目 skill

项目型产物建议结构：

```text
[repo-name]-skill/
├── SKILL.md
├── guides/
│   ├── 01-installation.md
│   └── 02-quickstart.md
└── troubleshooting.md
```

生成要求：

- 所有命令必须来自真实 README、文档、配置或源码入口。
- 不编造不存在的参数、环境变量或输出。
- guide 必须写清适用场景、可执行命令、预期结果和完成检查。
- troubleshooting 至少 3 条，极简项目至少 2 条。
- 项目过小时，不硬凑长文档，诚实说明能力边界。

可运行：

```bash
bash scripts/validate-project-skill.sh /path/to/generated-skill
```

如果分数或结构不达标，按 `references/project/project-distillation.md` 的质量门修复后再交付。

## 更新已有 Skill

更新前先读取旧版：

1. 保留用户已有的触发词、命名习惯和目录结构。
2. 标注新增信息来自哪里，尤其是最近动态或项目版本变化。
3. 修复过时内容时不要静默删除重要争议或旧边界。
4. 更新后重新检查占位符、引用路径和可执行脚本。

## 输出口径

完成后告诉用户：

- 新 skill 的实际路径。
- 合并/生成了哪些能力。
- 是否删除或停用了旧入口。
- 跑过哪些验证，哪些没跑成。
